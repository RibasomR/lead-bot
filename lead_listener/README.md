# 🎧 Lead Listener Service

## Описание

Lead Listener — это userbot-сервис для автоматического мониторинга Telegram-чатов и обнаружения потенциальных лидов по ключевым словам.

## Основные возможности

### ✅ Реализовано

**Мониторинг и фильтрация:**
- ✅ Управление несколькими Telethon клиентами одновременно (до 3-5 аккаунтов)
- ✅ CLI утилита для авторизации аккаунтов с поддержкой 2FA
- ✅ Автоматическое переподключение при разрыве связи
- ✅ Подписка на события новых сообщений в чатах
- ✅ Распределение чатов между аккаунтами (round-robin)
- ✅ Фильтрация сообщений по ключевым словам (регистронезависимая, мультиязычная)
- ✅ Определение языка сообщения (ru/en/other)
- ✅ Извлечение стека технологий из текста (50+ распознаваемых)
- ✅ Проверка на дубликаты (по chat_id + message_id)
- ✅ Игнорирование сообщений от ботов и системных

**Работа с лидами:**
- ✅ Автоматическое создание лидов в БД
- ✅ Асинхронный AI-анализ лидов
- ✅ Отправка уведомлений в Admin Bot
- ✅ Формирование прямых ссылок на сообщения

**Отправка сообщений:**
- ✅ HTTP API сервер (порт 8001) для приёма команд
- ✅ Отправка сообщений через userbot от выбранного аккаунта
- ✅ Поддержка reply_to для ответов на конкретные сообщения
- ✅ Обработка всех Telegram ошибок (FloodWait, ChatWriteForbidden, etc.)
- ✅ Логирование отправленных сообщений в БД

**Антиспам механизмы:**
- ✅ Счётчик откликов в час на каждый чат
- ✅ Проверка лимитов перед отправкой (HTTP 429 при превышении)
- ✅ Случайные задержки между отправками (2-10 секунд)
- ✅ Логирование всех попыток отправки (успешных и заблокированных)

**Инфраструктура:**
- ✅ Docker контейнеризация с портом 8001
- ✅ Graceful shutdown всех компонентов
- ✅ Healthcheck эндпоинты

## Использование

### Авторизация аккаунтов

Перед запуском основного сервиса необходимо авторизовать userbot-аккаунты:

```bash
# Локально
python lead_listener/auth_cli.py

# Или через Docker
docker-compose run --rm lead_listener python lead_listener/auth_cli.py
```

**Интерактивное меню:**
1. **Добавить новый аккаунт** — авторизация нового Telegram аккаунта
2. **Просмотреть существующие аккаунты** — список всех аккаунтов в БД
3. **Удалить аккаунт** — удаление аккаунта из системы

**Шаги авторизации:**
1. Введите номер телефона в международном формате (например, `+79991234567`)
2. Введите название аккаунта (например, "Рабочий 1")
3. Выберите стиль общения по умолчанию:
   - 1️⃣ Вежливый/деловой (polite)
   - 2️⃣ Неформальный/дружеский (friendly)
   - 3️⃣ Агрессивный/жёсткий (aggressive)
4. Введите код из Telegram
5. Если требуется — введите пароль 2FA

После успешной авторизации сессия сохраняется в `sessions/session_{tg_user_id}.session`.

## Антиспам механизмы

Lead Listener включает встроенные механизмы защиты от банов:

### Лимиты отправки

- **MAX_REPLIES_PER_CHAT_PER_HOUR** (по умолчанию: 5)
  - Максимум откликов в один чат за час
  - При превышении — HTTP 429

- **MIN_SEND_DELAY** / **MAX_SEND_DELAY** (по умолчанию: 2-10 сек)
  - Случайная задержка перед каждой отправкой
  - Имитирует человеческое поведение

### Обработка ошибок

- ✅ FloodWaitError — автоматическое ожидание
- ✅ ChatWriteForbiddenError — нет прав на отправку
- ✅ UserBannedInChannelError — аккаунт забанен
- ✅ MessageTooLongError — сообщение слишком длинное
- ✅ SlowModeWaitError — slow mode в чате

Все ошибки логируются в БД в таблицу `replies`.

### Запуск мониторинга

```bash
# Локально
python lead_listener/main.py

# Через Docker
docker-compose up lead_listener

# В фоновом режиме
docker-compose up -d lead_listener
```

### Логи

Логи сохраняются в:
- **Консоль:** цветной вывод с уровнем INFO
- **Файл:** `logs/leadhunter.log` с ротацией каждые 10 MB

Уровень логирования настраивается через `LOG_LEVEL` в `.env`.

## HTTP API

Lead Listener предоставляет HTTP API на порту 8001 для взаимодействия с Admin Bot.

### Эндпоинты

#### POST /api/send_message

Отправить сообщение в чат через userbot.

**Request:**
```json
{
  "lead_id": 123,
  "account_id": 1,
  "chat_tg_id": -1001234567890,
  "message_text": "Здравствуйте! Готов помочь.",
  "style_used": "friendly",
  "reply_to_message_id": 45678
}
```

**Response (success):**
```json
{
  "success": true,
  "message": "Message sent successfully"
}
```

**Response (rate limit exceeded):**
```json
{
  "success": false,
  "error": "Rate limit exceeded",
  "wait_seconds": 1234
}
```

#### GET /api/health

Проверка работоспособности сервиса.

```bash
curl http://localhost:8001/api/health
```

#### GET /api/accounts/status

Получение статуса всех аккаунтов.

```bash
curl http://localhost:8001/api/accounts/status
```

## Фильтрация лидов

### Критерии определения лида

Сообщение считается лидом, если:
1. ✅ Содержит одно или несколько ключевых слов
2. ✅ Не является системным/сервисным сообщением
3. ✅ Отправлено не ботом
4. ✅ Не дублирует существующий лид (по chat_id + message_id)

### Извлечение метаданных

Для каждого лида автоматически определяется:
- **Язык:** ru/en/other (по соотношению кириллицы и латиницы)
- **Стек технологий:** Python, aiogram, Next.js, PostgreSQL и т.д.
- **Информация об авторе:** ID, username, имя
- **Ссылка на сообщение:** для быстрого перехода

## Troubleshooting

### Ошибка: "Аккаунт не авторизован"

**Решение:** Запустите CLI для авторизации:
```bash
python lead_listener/auth_cli.py
```

### Ошибка: "FloodWaitError"

**Причина:** Telegram временно ограничил запросы.

**Решение:** Подождите указанное количество секунд. Система автоматически повторит попытку.

### Ошибка: "No active chats for monitoring"

**Причина:** В БД нет активных чатов для мониторинга.

**Решение:** Добавьте чаты через Admin Bot:
```
/add_chat
```

### Логи не появляются

**Проверка:**
1. Убедитесь, что папка `logs/` существует
2. Проверьте права доступа к папке
3. Проверьте переменную `LOG_FILE` в `.env`

### Сессии не сохраняются

**Проверка:**
1. Убедитесь, что папка `sessions/` существует
2. При использовании Docker проверьте volume mapping:
   ```yaml
   volumes:
     - ./sessions:/app/sessions
   ```
